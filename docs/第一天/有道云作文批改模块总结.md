# 有道云英语作文批改模块 - 面试场景题与开发总结

针对有道云作文批改模块（正式前后端分离接口 `EccController` 及其底层重构），以下是详细的技术考点与面试演练模板（包含技术细节与大白话解释）：

### 1. 你是如何设计这个模块的？
**【技术细节】**：
我采用了标准的 MVC 三层架构。为了适配前后端分离标准，Controller 层设计为 `@RestController` 并通过 `@RequestBody` 接收前端 JSON。为了统一公司各端的数据格式，我封装了泛型 `ApiResponse` 类。在对接外部 API 时，有道云返回的 JSON 结构较复杂，我没有使用脆弱的 `JSONObject` 去硬解析，而是设计了 `YoudaoResponse` 嵌套 DTO（数据传输对象），利用 Jackson 实现了强类型的反序列化。
**【大白话解释】**：
我把这个模块分得清清楚楚：Controller 就像是前台接待，专门负责收前端的包裹（JSON 数据）；Service 就像是跑腿的，拿着数据去跟有道云打交道。为了让前端兄弟好处理，我把所有返回结果都用一个叫 `ApiResponse` 的标准盒子装起来。有道云给的数据又多又乱，我专门建了个 Java 对象去接它，就像“对号入座”一样，这样以后代码维护起来特别稳，不会因为写错一个字符串键值就报错。

### 2. 你是如何实现这个模块的？
**【技术细节】**：
在实现上，我重点考虑了**健壮性**。首先在 Controller 层利用 Spring 的 `StringUtils.hasText` 对核心参数（如作文内容、年级、标题）做前置拦截，防止脏数据打穿到 Service 层。获取到第三方返回后，利用 Spring Boot 容器中的 `ObjectMapper` 实例进行解析。为防止有道云后续增加新字段导致反序列化崩溃，我在 DTO 上加了 `@JsonIgnoreProperties(ignoreUnknown = true)` 注解。所有的外部调用和解析都用 `try-catch` 包裹，异常时通过 `SLF4J` 记录带有 Trace 信息的日志，并给前端返回友好的系统降级提示。
**【大白话解释】**：
别人调我接口时，我第一步先严格检查他传的东西是不是空的或者全是空格。确认没问题了，才去请求有道云。拿到结果后，我用 Spring 自带的 Jackson 工具，自动把 JSON 字符串变成我们的 Java 对象。我还留了个心眼，告诉这个工具：“如果有道云以后加了新数据，你不认识的就直接跳过，别报错”。最后，如果中间网断了或者出错了，我会在后台偷偷拿小本本（log 日志）把错误证据记下来，然后和和气气地告诉前端“系统开小差了”，绝不把底层的乱码报错直接暴露给用户。

### 3. 你是如何测试这个模块的？
**【技术细节】**：
测试分为两个阶段。开发初期，我写了一个带有 `@RequestParam` 默认值的测试桩，快速验证了有道云 API 签名鉴权机制的连通性。正式模块完成后，我使用 Postman 构造 `application/json` 报文进行端到端测试。除了正常主流程，我重点做了**异常分支测试**：例如构造空参数测试防御逻辑；故意改错 AppKey 触发有道云报错，验证我的 JSON 解析器能否正确抓取 `errorCode`（或错误响应）并将错误信息包装进 `ApiResponse` 返回，确保系统不会抛出 HTTP 500。
**【大白话解释】**：
一开始为了图快，我先写了个“傻瓜式”的测试接口，不用填参数直接一跑，证明咱和有道云的网络和签名密码是通的。后来正式接口写好了，我就用 Postman 模拟真实前端，发 JSON 数据过去测。我不仅测了正常批改，还专门“找茬”：比如不传作文，或者故意把账号密码搞错，看看系统会不会崩溃。结果证明我的代码很坚挺，哪怕有道云那边报错了，我们这边也是稳稳地给前端回一条标准的提示，没出过洋相。

### 4. 这个项目的八股文（面试常考知识点）
**【技术细节】**：
1. **`@RestController` 和 `@Controller` 的区别？**
   答：`@RestController` 是 `@Controller` 和 `@ResponseBody` 的结合体。它告诉 Spring，这个类里所有方法的返回值都要直接写入 HTTP Response Body 中（通常转成 JSON），而不是去解析为视图（如 JSP/Thymeleaf）。
2. **`@RequestBody` 和 `@RequestParam` 的底层原理？**
   答：`@RequestBody` 触发 Spring 的 `HttpMessageConverter`（默认是 Jackson），读取 HTTP 请求体中的 IO 流，将其反序列化为 Java 对象。而 `@RequestParam` 只是从 URL 拼接参数或表单（`application/x-www-form-urlencoded`）中获取键值对。
3. **调用外部接口如何保证安全？（开放平台签名机制）**
   答：通常采用 Hash 摘要算法（如项目中用到的 SHA-256）。客户端将 `AppKey + 业务参数 + salt(随机盐) + 时间戳` 拼接后进行 Hash 生成签名（Sign）。服务端收到后用同样的规则计算比对。**加盐**防止彩虹表破解，**时间戳**防止抓包后进行“重放攻击”（Replay Attack）。

**【大白话解释】**：
面试官要是问你这块，你就记住这几个点：咱这接口是纯给数据的（不用渲染网页），所以用 `RestController`。前端传过来的是一个完整的 JSON 包裹，所以咱们用 `RequestBody` 去拆包；如果只是 URL 后面带的几个小参数，那才用 `RequestParam`。跟有道云打交道最怕别人截胡、篡改数据，所以我们每次请求都要算一个加密的签名，里面加上当前的时间和随机数，这就相当于盖了个“防伪且限时”的印章，黑客就算抓到了包，过一会儿这包也就作废了。
