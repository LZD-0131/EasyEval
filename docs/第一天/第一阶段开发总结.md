# 英语作文批改系统 - 第一阶段开发总结

## 一、 开发顺序与完成内容

在本项目的第一阶段（API 接入与核心链路打通阶段），我们主要完成了以下工作，开发顺序如下：

### 1. 探索与验证（API 可用性测试）
- **官方 Demo 运行**：首先定位并运行了项目中的有道云官方 Java 示例 (`EnTextCorrectWritingDemo.java`)。
- **凭证配置**：填入分配的 `appKey` 和 `appSecret`。
- **依赖问题解决**：解决了测试过程中遇到的 Maven 生命周期错误和本地仓库权限问题。
- **成功验证**：成功向有道云发送测试文本（"I is a good student. I very like english."），并获取到正确的批改建议（如 is->am, very->really），证明 API 接口完全可用。

### 2. Spring Boot 项目集成（核心业务层）
- **网络层依赖**：在主项目的 `pom.xml` 中引入 `okhttp3` 依赖，作为 HTTP 请求客户端。
- **工具类移植**：从 Demo 中提取并优化了核心工具类：
  - `AuthV3Util`：处理有道云 API V3 版本的 SHA-256 签名计算。
  - `HttpUtil`：封装 OkHttp 发起 POST 请求的逻辑。
- **配置分离**：将敏感信息（API Key、Secret）和请求 URL 抽离到 `application.properties` 配置文件中。
- **服务层封装**：创建 `YoudaoEssayEvalService`，通过 `@Value` 注入配置，对外提供统一的 `evaluateEnglishEssay` 方法。

### 3. Web 接口暴露与标准化（Controller 层）
- **初版 Controller**：创建 `EssayEvalController`，提供 `/api/eval/english` POST 接口，直接返回有道云的原始 JSON 字符串。
- **引入统一响应体**：为了规范企业级开发，仿照参考项目引入了 `ApiResponse<T>` 包装类（包含 status, message, data, timestamp）。
- **配置 Lombok**：为支持 `ApiResponse` 的 `@Data` 注解，在 `pom.xml` 中引入了 Lombok 依赖。

### 4. 数据模型映射（JSON 反序列化）
- **创建 DTO 实体**：针对有道云返回的复杂 JSON，创建了对应的 Java 实体类 `YoudaoResponse`。
  - 使用 `@JsonIgnoreProperties(ignoreUnknown = true)` 过滤不需要的冗余字段。
  - 使用 `@JsonProperty` 处理首字母大写的特殊字段（如 RequestId, Result）。
- **完善接口逻辑**：在 Controller 中注入 `ObjectMapper`，将有道云返回的 JSON 字符串优雅地反序列化为 `YoudaoResponse` 对象，再包裹进 `ApiResponse` 返回给前端。

## 二、 技术场景题与面试八股文

假设作为公司的面试官，针对第一阶段的项目实现，可能会问出以下技术场景题（附带详细技术细节的回答）：

### 1. 你是如何设计这个模块的？
- **回答思路（详细技术细节 + 大白话解析）**：
  - **架构分层设计**：严格遵循 MVC 规范，将职责划分为表现层（Controller）、业务逻辑层（Service）和工具层（Utils）。
    - `EssayEvalController` 仅负责 HTTP 请求的路由、参数校验（使用 `@RequestParam` 并设置默认值避免 NPE）以及结果的封装。
    - `YoudaoEssayEvalService` 负责核心的业务逻辑编排，包括准备 API 请求所需的参数（q, grade, title 等）。
    - `AuthV3Util` 和 `HttpUtil` 作为底层基础设施，负责与有道云进行纯粹的签名计算和网络 I/O 交互。
    - 💡 **大白话**：*就是“各司其职，不乱成一锅粥”。Controller 像前台接待员，只负责收发文件；Service 像车间主任，负责把活儿派好；Utils 像底层工人，只管埋头干具体的网络请求和加密这种脏活累活。*
  - **配置解耦设计**：为了避免硬编码带来的安全风险和多环境部署问题，我将有道云的 `appKey` 和 `appSecret` 配置在 `application.properties` 中，在 Service 层通过 Spring 的 `@Value("${youdao.api.key}")` 动态注入。
    - 💡 **大白话**：*就是“密码不能写死在代码里”。写死的话，代码一开源密码就泄露了，而且如果以后换了账号，还得去改代码、重新打包。写在配置文件里，只要改个配置重启就行了。*
  - **接口协议设计**：设计了全局统一响应实体 `ApiResponse<T>`，包含状态码（status）、提示信息（message）、时间戳（timestamp）和泛型数据载荷（data）。这使得前端在处理成功或异常时有一套标准的解析逻辑，符合企业级 RESTful API 的设计规范。
    - 💡 **大白话**：*就是“给前端兄弟行个方便”。不管后台发生啥，我都给你一个固定格式的包裹。包裹外面写着状态（成功还是失败），里面才是真正的数据。这样前端闭着眼睛都能拆包，不会因为报错了就整个页面崩溃。*
  - **数据传输设计 (DTO)**：由于有道云返回的 JSON 极其庞大且嵌套较深（包含 `Result`, `majorScore`, `essayFeedback` 等多层结构），我设计了 `YoudaoResponse` 及其静态内部类来精确映射需要的数据。通过 `@JsonIgnoreProperties(ignoreUnknown = true)` 忽略了不需要的冗余字段，减少了内存开销，并通过 `@JsonProperty` 解决了 API 返回字段首字母大写不符合 Java 驼峰命名规范的问题。
    - 💡 **大白话**：*有道云扔过来的数据像个大垃圾堆，什么都有，而且字段名还乱七八糟（首字母大写）。我专门建了一个模具（DTO），只挑我们要的（总分、建议、错误详情）装进去，多余的直接扔掉，然后把名字也统一改成 Java 习惯的驼峰命名。*

### 2. 你是如何实现这个模块的？
- **回答思路（详细技术细节 + 大白话解析）**：
  - **鉴权机制实现**：有道云 API V3 要求严格的请求签名机制以防篡改。我实现了 SHA-256 哈希算法，具体逻辑是将 `appKey` + 截断后的输入文本（前10后10个字符中间加长度） + `salt`（UUID 生成） + `curtime`（秒级时间戳）+ `appSecret` 拼接成字符串，然后利用 Java 自带的 `MessageDigest.getInstance("SHA-256")` 计算出摘要，最后转换为十六进制字符串作为 `sign` 参数。
    - 💡 **大白话**：*就是“怎么证明我是我，而且没人动过我的数据”。我把账号、密码、当前时间、随机数和我的作文内容，按照有道云规定的公式搅拌在一起，算出一个独一无二的“指纹”。有道云收到后也算一遍，如果指纹对得上，才给我批改。*
  - **网络请求实现**：为了提高并发性能和连接复用率，我放弃了 JDK 原生的 `HttpURLConnection`，选用了成熟的 `OkHttp 4.10.0` 客户端。通过构建 `FormBody` 将参数封装为 `application/x-www-form-urlencoded` 格式，然后使用 `OkHttpClient` 发起同步的 POST 请求。为了防止请求挂起，可以在初始化 client 时设置合理的 connectTimeout 和 readTimeout。
    - 💡 **大白话**：*Java 自带的网络请求工具太老太慢了，我换成了大厂都在用的 OkHttp。它底层有连接池，发请求就像走高速公路，不用每次都停下来交过路费（握手），速度快很多。另外我还设置了超时时间，万一有道云那边卡死了，我这边也不会一直傻等。*
  - **JSON 序列化与反序列化实现**：在获取到 OkHttp 返回的 `Response` 对象后，提取其 `body().string()`。由于需要在 Controller 中返回结构化的数据，我通过 Spring Boot 自动装配的 Jackson `ObjectMapper`，调用 `objectMapper.readValue(jsonStr, YoudaoResponse.class)`，利用反射机制将 JSON 字符串精准地反序列化为我们设计的 DTO 对象，最后通过 `ApiResponse.success(data)` 包装返回给前端。
    - 💡 **大白话**：*有道云返回的只是一长串纯文本（字符串）。为了方便我们在代码里操作（比如拿分数、拿评语），我用 Spring 自带的 Jackson 工具，把这串长文本“变魔术”一样直接转换成了我们写好的 Java 对象。最后再装进我们的标准包裹里发给前端。*

### 3. 你是如何测试这个模块的？
- **回答思路（详细技术细节 + 大白话解析）**：
  - **第一步：环境隔离与签名验证（单元级）**：首先在不启动 Spring 容器的情况下，直接编写一个包含 `main` 方法的独立 Java 类。硬编码填入 `appKey` 和 `appSecret`，发送一条简单的测试文本 "I is a good student."。通过打印的日志验证 SHA-256 签名是否计算正确，以及能否成功拿到 `errorCode: 0` 的响应，以此证明鉴权逻辑和网络连通性没有问题。
    - 💡 **大白话**：*“先在沙盒里玩”。不启动整个大项目，只写个很小的 main 方法跑一下，主要是看看有道云的账号密码对不对，加密算法有没有写错，能不能调通。这一步保证了核心的轮子没问题。*
  - **第二步：Spring 容器依赖与集成测试**：在将代码迁移到 Spring Boot 的组件中后，我使用 Maven 命令 `mvn spring-boot:run` 启动了内嵌的 Tomcat 容器（配置在 8081 端口以避免冲突）。
    - 💡 **大白话**：*把轮子装到车上。把刚才写好的工具类放进 Spring Boot 框架里，把服务跑起来，看看有没有端口冲突或者依赖报错。*
  - **第三步：端到端（E2E）接口验证**：为了验证 Controller 层的路由和 JSON 反序列化是否正常工作，我使用 `curl` 命令行工具模拟真实客户端发起 POST 请求：`curl.exe -X POST "http://localhost:8081/api/eval/english" -d "text=I very like apple." -d "grade=high"`。通过观察终端输出的完整 JSON 响应，确认了中文字符没有乱码（UTF-8 编码正常），`ApiResponse` 结构封装正确，且内部的 `YoudaoResponse` 成功将有道云的数据映射为了 JSON 对象而不是转义字符串。
    - 💡 **大白话**：*“假装自己是前端”。我用命令行工具直接去敲我们写好的接口，看返回的数据是不是我们刚才设计的那个标准包裹，里面中文有没有变成乱码，JSON 是不是干净清爽的。没问题了，才敢把接口交给前端兄弟。*

### 4. 这个项目的八股文
- **详细技术考点（附大白话）**：
  - **Spring 原理类**：
    - `@Autowired` 和 `@Resource` 的区别？（`@Autowired` 是 Spring 提供的，默认按类型(byType)注入；`@Resource` 是 JDK 提供的，默认按名称(byName)注入）。
      - 💡 **大白话**：*`@Autowired` 是说“给我找个会修车的人”，谁会修车就让谁来；`@Resource` 是说“把叫王师傅的人给我找来”，指定要这个人。*
    - `@RestController` 是如何将对象转为 JSON 的？（它是一个复合注解，包含了 `@Controller` 和 `@ResponseBody`。`@ResponseBody` 会触发 Spring MVC 的 `HttpMessageConverter` 机制，默认使用 `MappingJackson2HttpMessageConverter` 调用 Jackson 库将返回的 Java 对象序列化为 JSON 写入 HTTP 响应体中）。
      - 💡 **大白话**：*加了这个注解，Spring 就像请了一个自动翻译官。我们只管丢给它 Java 对象，翻译官会在最后出门前，自动把对象翻译成前端认识的 JSON 格式。*
  - **Java 核心与并发类**：
    - OkHttp 的底层连接池 (`ConnectionPool`) 是如何工作的？为什么比 `HttpURLConnection` 性能好？（OkHttp 默认维护一个连接池，支持 HTTP/2 多路复用，通过复用底层的 TCP Socket 减少了三次握手和四次挥手的开销）。
      - 💡 **大白话**：*传统的网络请求每次都要打电话、问好、挂电话（三次握手四次挥手），很费时间。OkHttp 搞了个“专线”，电话打通了就不挂，下次有事直接说，省去了每次拨号的时间，所以快很多。*
    - SHA-256 算法生成的字符串长度是多少？（256 位，即 32 字节，转换为十六进制字符串后长度固定为 64 个字符）。
  - **网络与安全类**：
    - 有道云 API 的鉴权机制中，为什么要加入 `salt`（随机数 UUID）和 `curtime`（时间戳）？（加入时间戳是为了防范**重放攻击**，服务端会校验时间戳是否过期；加入随机盐和参数混合计算 SHA-256，是为了验证请求者的身份，同时保证请求参数在传输过程中**没有被中间人篡改**，因为一旦参数改变，算出的 sign 就会与服务端算出的不一致）。
      - 💡 **大白话**：*加时间戳是为了防止“黑客录音”。黑客把你刚才发给服务器的加密请求原封不动截获，过一会再发一遍（重放攻击），服务器一看时间过期了就不认了。加随机数和参数混合加密，是为了防止“中途掉包”，别人如果在半路把作文内容改了，最后算出来的指纹就对不上了，服务器直接拒收。*
  - **JSON 序列化类**：
    - 为什么要在 DTO 类上加 `@JsonIgnoreProperties(ignoreUnknown = true)`？（如果不加这个注解，当第三方 API 返回的 JSON 中新增了某些字段，而我们的 Java 实体类中没有对应的属性时，Jackson 在反序列化时会抛出 `UnrecognizedPropertyException` 异常导致请求失败。加上此注解可以提高系统的健壮性，忽略未知的冗余字段）。
      - 💡 **大白话**：*有道云哪天万一抽风，偷偷在数据里多加了一个叫“版本号”的字段，如果我们没加这个注解，代码一看到不认识的字段直接就崩溃罢工了。加了这个注解，就是告诉代码：“不认识的字段直接无视掉”，系统就不会莫名其妙崩了。*
