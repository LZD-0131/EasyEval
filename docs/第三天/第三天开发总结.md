# 第三天开发总结：用户查看卡券功能打通

## 面试官场景模拟 Q&A

### 模块一：用户查看卡券（openId → userId → 卡券列表）（第一件事）

#### 1. 你是如何 design 这个模块的？
**👨‍💻 技术细节：**
我将整个能力拆成三段链路：**前端获取 code → 后端换取 openId → 数据库查询 userId → 查询卡券并组装响应**。  
控制层以 [CouponController](file:///e:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/controller/CouponController.java) 为核心，使用 `WxMpService` 通过 OAuth2 code 换取 `openId`，再通过 `t_user.wechat_user_id` 找到 `userId`，最终查询 `t_coupon` 并补充 `t_coupon_type` 的类型信息，输出 `CouponResp` 列表。  
前端 [coupon.html](file:///e:/SpingbootStudy/easyeval/src/main/resources/static/coupon.html) 只负责从 URL 里取 code，再向后端 `GET /api/mycoupon` 拉取数据并渲染页面，从而保持前后端职责清晰。

**🗣️ 大白话：**
我把流程拆成“拿身份证 → 找人 → 查卡券”。  
前端只负责拿到微信给的 code，后端把 code 交给微信换 openId，再用 openId 去数据库里找这个人，然后再查他有哪些卡券，最后把结果回给页面显示。

#### 2. 你是如何实现这个模块的？
**👨‍💻 技术细节：**
1. **OAuth2 授权链接处理**：通过 [HEllo](file:///e:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/controller/HEllo.java) 提供 `/getUrl` 接口，使用 `wxMpService.getOAuth2Service().buildAuthorizationUrl(...)` 生成授权链接（scope 为 `snsapi_base`）。
2. **后端取 openId**：在 [CouponController](file:///e:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/controller/CouponController.java#L47-L74) 中调用 `wxMpService.getOAuth2Service().getAccessToken(code)`，获取 `openId`。
3. **openId → userId → 卡券数据**：用 `wechat_user_id` 查询 `TUser`，再用 `user_id` 查询 `TCoupon`，最后补充 `TCouponType` 形成 `CouponResp`。
4. **前端渲染**：在 [coupon.html](file:///e:/SpingbootStudy/easyeval/src/main/resources/static/coupon.html) 中读取 `code` 参数，调用 `https://11ir9803988vm.vicp.fun/api/mycoupon` 获取列表并渲染。

**🗣️ 大白话：**
我先让系统会“造授权链接”，再让后端会“换 openId”，接着用 openId 找人、找卡券。  
前端页面只负责发请求和展示结果，不做业务判断。

#### 3. 你是如何测试这个模块的？
**👨‍💻 技术细节：**
1. 在公众号菜单里点击“我的卡券”，触发 OAuth2 链接，微信会携带 `code` 回到 `coupon.html`。  
2. 打开浏览器 Network 查看 `GET /api/mycoupon?code=xxx` 是否成功返回。  
3. 如果返回 200 且 `data` 有数组，说明 `openId → userId → 卡券` 链路打通。  

**🗣️ 大白话：**
点菜单，看页面有没有请求接口。如果接口返回了卡券列表，说明流程就通了。

#### 4. 这个项目的八股文（底层原理拓展）
**👨‍💻 技术细节：**
- **为什么一定要用 OAuth2 code 换 openId？**  
  因为微信只给前端一个一次性的 `code`，真正能识别用户的 `openId` 必须通过后台调用微信接口换取，避免前端伪造。
- **`snsapi_base` 与 `snsapi_userinfo` 的区别？**  
  `snsapi_base` 静默授权，只能拿 openId；`snsapi_userinfo` 需要用户确认，可拿昵称头像。
- **为什么 Controller 要注入 `WxMpService` 而不是静态工具类？**  
  Spring Bean 便于管理配置、测试与替换；静态类无法享受 IoC 容器带来的生命周期管理。

**🗣️ 大白话：**
- code 就像一次性取号纸，必须去微信窗口换成真正的身份（openId）。  
- 静默授权省事但信息少，强授权信息多但要用户点同意。  
- 用 Spring 的 Bean，就像把工具交给管家统一管理，后续升级更省心。

## 管理端 IP 鉴权功能实现（第七件事）

### 1. 为什么要实现 IP 鉴权？
**👨‍💻 技术细节：**
对于敏感的管理端接口（如 `/api/admin/pay` 充值接口），仅靠业务逻辑校验是不够的。为了防止接口被恶意调用，我们通过 **Servlet Filter (过滤器)** 在请求进入控制器之前进行拦截，校验调用方的 IP 地址是否在白名单内。这属于**网络层级的访问控制**。

**🗣️ 大白话：**
给管账的接口加个“看门大爷”。只有拿着管理员家钥匙（指定 IP）的人才能进来充值，外人哪怕知道接口地址也进不来。

### 2. IP 过滤器的底层实现
**👨‍💻 技术细节：**
1. **过滤器定义**：编写 [IpFilter](file:///e:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/fillter/IpFilter.java)，实现 `jakarta.servlet.Filter` 接口。
2. **IP 获取逻辑**：由于生产环境可能存在代理（如 Nginx、花生壳），代码中通过 `X-Forwarded-For`、`X-Real-IP` 等 Header 逐级获取真实的客户端 IP，最后兜底使用 `request.getRemoteAddr()`。
3. **注册与匹配**：在 [FilterConfig](file:///e:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/config/FilterConfig.java) 中使用 `FilterRegistrationBean` 进行注册，并通过 `addUrlPatterns` 精确指定拦截路径。

**🗣️ 大白话：**
写一个过滤网，先想办法查出对方真正的电脑地址（IP），然后跟白名单比对。如果对不上，直接回一句“Access denied”，把请求扔出去。

### 3. 如何测试该功能？
**👨‍💻 技术细节：**
1. **白名单测试**：将 [IpFilter](file:///e:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/fillter/IpFilter.java#L20) 中的 `allowedIpAddress` 修改为你当前的实际公网 IP。访问接口，应返回正常的业务数据。
2. **非法访问测试**：随便改动一个数字（使白名单失效），再次访问接口。浏览器或 Postman 应收到字符串：`Access denied for your IP address.`。

**🗣️ 大白话：**
把名单改成你的名字，看能不能进；把名单改错，看是不是被拦在门外。

## 公众号回复故障修复与路由优化（第五件事）

### 1. 为什么之前后台不报错但公众号不回复？
**👨‍💻 技术细节：**
1. **配置标识缺失**：`WxMpMessageRouter` 默认寻找名为 `default` 的配置。我们在 [WxMpConfig](file:///e:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/config/WxMpConfig.java) 中通过 `service.setMultiConfigStorages` 显式注册了 `"default"` 和真实的 `appId`，解决了路由定位失败的问题。
2. **异步处理超时**：微信要求服务器在 5 秒内同步响应。我们在 [WxResponse](file:///e:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/controller/WxResponse.java) 的路由规则中统一添加了 `.async(false)`，确保消息处理是同步且即时的。

**🗣️ 大白话：**
之前就像收件人名字写错了（没标 default），邮递员（Router）找不到要把信给谁。我们把名字补齐了，并且规定必须“当面签收、立刻回信”（同步处理），公众号就恢复正常回复了。

## Redis 分布式锁实现与幂等性保障（第六件事）

### 1. 为什么要在这里加分布式锁？
**👨‍💻 技术细节：**
为了防止用户在弱网环境下快速多次点击“提交批改”按钮，导致系统产生多条重复记录、多次扣减用户额度，甚至多次调用昂贵的第三方 API。这在分布式系统中被称为**幂等性控制**。

**🗣️ 大白话：**
防止用户手抖点快了。就像坐公交扫码，1秒钟扫两次只能算一个人上车，不能扣两份钱。

### 2. 分布式锁的底层实现原理
**👨‍💻 技术细节：**
1. **连接池适配**：在 [JedisConfig](file:///e:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/config/JedisConfig.java) 中优化了构造逻辑，自动识别 `username` 是否为空，解决了 Redis 6.0+ 认证导致的 `WRONGPASS` 报错。
2. **原子性加锁**：在 [RedisTool](file:///e:/SpingbootStudy/easyeval/eaasyeval/src/main/java/com/javacoderpro/eaasyeval/utils/RedisTool.java) 中使用 `jedis.set(key, value, params)` 命令，配合 `NX`（不存在则设置）和 `EX`（过期时间）参数。这是一个**原子操作**，保证了锁的互斥性和自动释放。
3. **业务拦截**：在 [EccController](file:///e:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/controller/EccController.java#L64-L68) 提交作文的最前端，以 `user_code_ + code` 作为唯一锁 Key。如果 5 秒内同一个 code 再次请求，直接拦截并返回“操作频繁”。

**🗣️ 大白话：**
当有人提交作文时，先去 Redis 里占个坑（加锁），有效期 5 秒。如果这 5 秒内还有人用同一个 code 进来，发现坑被占了，就只能在外面等着，不让他进屋（执行业务）。

### 3. 面试官场景题：如果 Redis 挂了怎么办？
**👨‍💻 技术细节：**
- **Fail-safe（失效安全）**：目前的实现中，如果 Redis 报错，会导致业务流程中断。在生产环境下，可以考虑“降级策略”，例如 Redis 不可用时退化为本地缓存锁（如 Guava Cache）或者直接放行并依靠数据库唯一索引兜底。
- **锁释放问题**：通过 `EX` 参数设置过期时间，避免了因后端程序崩溃导致的“死锁”问题。

**🗣️ 大白话：**
如果 Redis 坏了，现在系统会报错。以后可以优化成：如果 Redis 坏了，就先用电脑内存记一下，或者干脆直接让用户过去，别耽误人家用，虽然风险高点但体验好。

## 面试高频“八股文”扩展（技术深度）

### 1. 微信 OAuth2 授权中的 code 为什么只能用一次？
**👨‍💻 技术细节：**
这是为了防止**授权重放攻击**。`code` 是用户授权的临时凭证，如果 `code` 可以多次使用，攻击者拦截到 `code` 后就可以反复换取用户的 `access_token`。微信设计为一次性使用且有效期极短（5分钟），并要求后端必须配合 `AppSecret` 才能换取 `openId`，形成闭环验证。

**🗣️ 大白话：**
`code` 就像电影票根，检过票进场后就作废了。这样即使别人捡到这张票根，也没法再进场，保证了安全性。

### 2. Spring 事务失效的场景有哪些？（结合项目实战）
**👨‍💻 技术细节：**
我们在 [EccController](file:///e:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/controller/EccController.java) 中使用了 `@Transactional`。面试官常问失效场景：
1. **非 public 方法**：Spring AOP 默认只拦截公开方法。
2. **类内部调用**：本类方法 A 调用本类带事务的方法 B，事务不会生效（绕过了代理对象）。
3. **异常被 catch 了**：如果代码里自己 `try-catch` 了异常且没抛出，Spring 无法感知异常，就不会回滚。
4. **数据库引擎不支持**：比如使用了 MyISAM 引擎而非 InnoDB。

**🗣️ 大白话：**
事务就像保险。如果你把保险锁在私人抽屉里（非 public）、自己人私下转账（类内部调用）、或者出事了瞒着保险公司（手动 catch 异常），那这保险就起不了作用。

### 3. 为什么选择雪花算法（Snowflake）生成 userId？
**👨‍💻 技术细节：**
相比 UUID，雪花算法生成的 ID 是**趋势递增**的长整型（Long）。在 MySQL 中作为主键，可以保持 B+ 树索引的有序性，减少页分裂和随机 IO，提升写入性能。
相比自增 ID，雪花算法在**分布式环境**下不需要依赖数据库，可以在应用层生成，避免了数据库单点压力和分库分表后的 ID 冲突问题。

**🗣️ 大白话：**
UUID 是一堆乱码，存数据库很慢；自增 ID 在人多的时候容易撞车。雪花算法就像是给每个人发一个带时间戳的排队号，既有序又不冲突，速度还快。

### 4. 微信临时素材与永久素材的区别及应用场景？
**👨‍💻 技术细节：**
- **临时素材**：保存 3 天，不占公众号配额，适合图片、语音等高频变动的内容。使用 `mediaUpload` 接口。
- **永久素材**：长期保存，图片上限 5000 个，适合菜单回复、图文消息中的常驻图片（如收款码）。使用 `materialFileUpload` 接口。

**🗣️ 大白话：**
临时素材像快餐盒，用完就扔，不占地儿；永久素材像家里的瓷碗，洗洗能一直用，但橱柜（微信配额）空间有限，要省着点用。

## code → openId → userId 链路补充说明（必须掌握）
**👨‍💻 技术细节：**
1. 前端从 OAuth2 授权链接进入，微信会把一次性 `code` 带回页面；
2. 后端用 `code` 调用微信接口换取该用户的唯一标识 `openId`；
3. 后端根据 `openId` 在 `t_user` 表中查询，定位到系统内部的 `userId`；
4. 拿到 `userId` 后，即可进行后续的业务操作（如查卡券、查批改记录、扣减额度等）。

**🗣️ 大白话：**
`code` 是一次性的取号纸，`openId` 是微信颁发的身份证，`userId` 是我们家账本上的内部编号。  
只有把这三者链路打通，我们才知道“当前是谁在干活”。

## 作文批改提交链路补充总结（第二件事）

### 1. code、openId、userId 的绑定与调用链路（重点）
**👨‍💻 技术细节：**
1. 前端页面 [eaasyeval.html](file:///e:/SpingbootStudy/easyeval/src/main/resources/static/eaasyeval.html) 通过 OAuth2 授权链接进入，地址栏携带 `code` 参数；前端脚本会自动读取 `code` 并随作文数据一起提交到 `POST /api/ecc/submit`。
2. 后端 [EccController](file:///e:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/controller/EccController.java) 用 `wxMpService.getOAuth2Service().getAccessToken(code)` 换取 `openId`。
3. 使用 `openId` 查询 `t_user.wechat_user_id`，定位到系统内部唯一的 `userId`，然后再用 `userId` 查询 `t_correction_count` 作为计数依据。
4. 只有当 `userId` 成功匹配且还有剩余次数时，才允许继续调用有道接口并返回批改结果。

**🗣️ 大白话：**
用户点菜单进页面时，微信会偷偷给一个一次性的 `code`。  
后端把这个 `code` 交给微信换成真正身份（`openId`），再用 `openId` 去数据库里找人，拿到系统里的 `userId`。  
只有找到这个人，系统才知道“这是谁在提交作文”，也才能扣次数和保存记录。

### 2. 事务为什么必须加（重点）
**👨‍💻 技术细节：**
在一次成功批改后，系统需要同时完成两件事情：  
1) 插入一条 `t_correction_request` 记录；  
2) 把 `t_correction_count.used_count` 自增 1。  
这两个操作属于同一次业务动作，必须保证原子性。所以在 [EccController](file:///e:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/controller/EccController.java) 的 `submit` 方法上加了 `@Transactional`。  
并且更新次数时改为按 `user_id` 精确更新（`setSql("used_count = used_count + 1")`），避免主键缺失导致更新失败但不报错。

**🗣️ 大白话：**
“保存批改记录”就像打印小票，“扣次数”就像刷卡扣钱。  
如果没有事务，可能出现“打印了小票但没扣钱”的情况。  
加事务就是告诉数据库：**这两件事必须一起成功，否则全部回滚**，从而保证数据一致。

### 3. 问题现象与最终修复
**👨‍💻 技术细节：**
现象是“批改成功，但次数未扣减”。  
最终修复方式是：  
1) 用事务包裹“写记录 + 扣次数”；  
2) 更新次数使用按 `user_id` 的 SQL 自增更新，避免 `updateById` 因主键为空导致更新无效。  
修复后，`t_correction_request` 会新增记录，`t_correction_count.used_count` 正常 +1。

**🗣️ 大白话：**
之前是“显示成功了，但账没扣”。  
现在把两个动作绑死在一起，谁都别想漏扣，结果就正常了。

## 用户查看批改记录功能实现（第三件事）

### 1. 功能目标与页面入口
**👨‍💻 技术细节：**
通过 OAuth2 授权链接跳转到 [evalrecode.html](file:///e:/SpingbootStudy/easyeval/src/main/resources/static/evalrecode.html)，前端读取 `code` 后调用后端 `POST /api/ecc/eval_record`，返回当前用户的批改记录列表。

**🗣️ 大白话：**
用户点菜单进入“批改记录”，页面自动带着 `code` 请求后端，拿到自己的记录列表并展示。

### 2. 后端接口实现逻辑
**👨‍💻 技术细节：**
接口位置在 [EccController](file:///e:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/controller/EccController.java#L116-L147)。核心流程：
1. 校验 `code` 非空。
2. 通过 `wxMpService.getOAuth2Service().getAccessToken(code)` 获取 `openId`。
3. 使用 `openId` 查询 `t_user.wechat_user_id` 获取 `userId`。
4. 用 `userId` 查询 `t_correction_request` 列表并按 `updated_at` 倒序返回。

**🗣️ 大白话：**

## 异常处理与移动端兼容性优化（第八件事）

### 1. 405 Method Not Allowed 错误排查
**👨‍💻 技术细节：**
在调试管理端充值接口 `/api/admin/pay` 时，用户直接在地址栏输入 URL 导致报错。这是因为地址栏发起的是 `GET` 请求，而后端 [PayController](file:///e:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/controller/PayController.java) 定义的是 `@PostMapping`。  
**解决方案**：必须通过 [pay.html](file:///e:/SpingbootStudy/easyeval/src/main/resources/static/pay.html) 页面提交表单，由 JavaScript 的 `fetch` 函数发起真正的 `POST` 请求。

**🗣️ 大白话：**
就像银行的“存款窗口”不给查余额一样。你在浏览器地址栏直接敲网址是“查”，后端规定这个地址只能“存（POST）”，所以会把你拒之门外。

### 2. 40163 错误：微信 code 只能用一次
**👨‍💻 技术细节：**
在卡券兑换成功后，如果执行 `location.reload()` 刷新页面，浏览器会带着旧的 `code` 再次请求后端，导致微信报错 `40163 (code been used)`。  
**解决方案**：优化 [coupon.html](file:///e:/SpingbootStudy/easyeval/src/main/resources/static/coupon.html) 的交互逻辑，兑换成功后通过 JavaScript **局部更新 UI**（修改按钮文字为“已兑换”并禁用），不再触发页面刷新，从而避开 code 的重复消耗。

**🗣️ 大白话：**
`code` 就像一张一次性电影票。你进场（加载列表）的时候已经检过票了，如果你出来再想进去（刷新页面），检票员（微信）就会告诉你这张票用过了。所以我们现在改成了：进场后就在里面待着，办完事直接改状态，不重新检票。

### 3. 移动端兼容性与跨域排查
**👨‍💻 技术细节：**
在真机测试时，发现“兑换”按钮没反应。排查发现两个核心点：
1. **域名硬编码**：前端 `fetch` 路径写死了公网域名，在不同网络环境下触发了跨域拦截。
2. **字段不匹配**：后端返回的是 `status: 200`，前端判断的是 `code === 200`，导致逻辑走空。  
**解决方案**：将前端所有接口调用改为**相对路径**（如 `/api/exchange`），并统一判定 `result.status === 200`。同时引入 `vConsole` 辅助真机调试（验证后已移除）。

**🗣️ 大白话：**
手机端比电脑端更“挑剔”。如果地址写死或者是回执单上的暗号对不上，手机就直接罢工。我们把地址改成“自家的（相对路径）”，把暗号对齐，手机端就顺畅了。

## RocketMQ 支付完成解耦与幂等消费实现（新增）

### 1. 你是如何设计这个模块的？
**技术细节：**
这次我把“管理员充值成功后的后续处理”从同步直调改成了事件驱动。原来 [PayController.java](E:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/controller/PayController.java) 里既要保存支付记录，又要直接发券，这样支付流程和发券流程耦合太紧，后面如果继续新增“送积分、发通知、记日志”等需求，控制器会越来越重。

所以这里引入 RocketMQ，把“支付完成”抽象成一条业务事件，主题是 `PAY_FINISH`。支付接口只负责写入 `t_payment_history` 并发送一条支付完成消息，发券逻辑放到 [MessageConsumer.java](E:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/consumer/MessageConsumer.java) 中异步消费。这样后续如果要新增优惠券、积分、通知、审计，只需要增加消费者，而不需要反复改支付主流程。

另外，这里统一采用的是 `rocketmq-client-java 5.0.6` 的 Producer + PushConsumer 模式，而不是旧版 `RocketMQTemplate` 和 `@RocketMQMessageListener`。这样和腾讯云 RocketMQ 5.x 的接入方式保持一致。

**大白话：**
以前像是“收银员一边收钱，一边自己跑去仓库发奖品”，后面仓库业务一多，收银台就乱了。现在改成“收银员收完钱后只广播一句：这个用户付钱了”，优惠券系统自己去领任务发券。支付和发券分开以后，后面加功能就更轻松了。

### 2. 你是如何实现这个模块的？
**技术细节：**
1. 在 [application.properties](E:/SpingbootStudy/easyeval/src/main/resources/application.properties) 中新增了 `rocketmq5.enabled/endpoints/access-key/secret-key/namespace/topic/tag/consumer-group` 配置，使用腾讯云公网接入地址连通 RocketMQ。
2. 在 [RocketMq5Config.java](E:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/config/RocketMq5Config.java) 中创建两个核心 Bean：
   - `Producer`：负责发送支付完成消息；
   - `PushConsumer`：负责订阅 `PAY_FINISH` 主题和 `PAY_FINISHED` 标签。
3. 在 [PayController.java](E:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/controller/PayController.java) 中：
   - 先保存 `TPaymentHistory`；
   - 再构造 `PayFinishedMessage`；
   - 最后发送 RocketMQ 消息。
4. 新增了 [PayFinishedMessage.java](E:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/dto/PayFinishedMessage.java)，消息里不只传 `userId`，而是同时携带 `paymentId`、`userId`、`paymentStatus`，让消费端拥有明确的业务主键。
5. 在 [MessageConsumer.java](E:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/consumer/MessageConsumer.java) 中收到消息后：
   - 先将消息体反序列化成 `PayFinishedMessage`；
   - 再根据 `paymentId` 生成固定券码 `PAY-<paymentId>`；
   - 先查 `t_coupon` 里是否已经存在这张券；
   - 如果不存在才真正插入优惠券；
   - 如果已存在则直接返回 `SUCCESS`，避免重复发券。

并且为了方便联调，我还在消费者中加入了日志：
- `Receive pay finished message...`
- `Skip duplicate coupon issue...`
- `Coupon issued successfully...`

**大白话：**
实现上就是两步：先发“通知”，再收“通知”。支付接口负责发通知，消费者负责收通知并发券。为了防止同一条通知被系统重复送过来，我给每一笔支付都带上唯一的 `paymentId`，然后把券码固定成 `PAY-支付ID`。这样只要数据库里已经有这张券，就说明这笔支付已经发过券了，后面再收到同样消息也不会重复送。

这部分我做了三层验证：

### 4. 这个项目在这里的八股文和面试场景题

#### 4.1 为什么要引入消息队列，而不是在支付接口里直接发券？
**技术细节：**
直接在支付接口里发券，优点是简单，但缺点是耦合严重、扩展性差、失败隔离差。支付主流程本质上只应该关心“支付是否成功”，发券、加积分、发通知这些都属于支付完成后的衍生动作，更适合异步解耦。

消息队列在这里主要解决三个问题：
1. **解耦**：支付模块不依赖具体的优惠券模块实现。
2. **异步**：支付成功后不需要等待发券逻辑跑完再返回。
3. **扩展**：后续新增积分、通知、审计时，只需要新增消费者。

**大白话：**
支付接口应该像前台收银，只负责收钱和记账；发券、送积分这些像后台仓库和运营，不应该全挤在收银台一起干。用了 MQ 后，收银台只管说一句“这笔钱付成功了”，后面谁需要这条消息谁自己处理。

#### 4.2 RocketMQ 在这个项目里具体承担什么角色？
**技术细节：**
RocketMQ 在这里承担的是“支付完成事件总线”的角色。生产者是 [PayController.java](E:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/controller/PayController.java)，消费者是 [MessageConsumer.java](E:/SpingbootStudy/easyeval/src/main/java/com/dong/easyeval/consumer/MessageConsumer.java)。现在只有一个消费者负责发券，但未来完全可以扩展多个消费者，比如积分服务、站内信服务、数据分析服务、财务审计服务。

**大白话：**
RocketMQ 就像公司里的广播系统。支付成功以后，系统会大喇叭喊一句“某某用户支付成功了”。优惠券部门听到了就发券，积分部门听到了就送积分，谁需要谁来接。

#### 4.3 为什么幂等性这么重要？
**技术细节：**
消息队列系统天然要面对“至少一次投递”的问题。也就是说，消费者可能会因为网络抖动、ACK 超时、应用重启等原因重复收到同一条消息。如果消费逻辑不做幂等控制，就可能出现同一笔支付发多张券的问题。

本项目里的幂等策略是：
1. 生产者发送消息时带上 `paymentId`；
2. 消费者用 `paymentId` 生成固定业务唯一键 `coupon_code = PAY-<paymentId>`；
3. 消费前先按这个业务唯一键查库；
4. 查到即跳过，查不到才创建。

这种方案本质上是“用业务唯一键防止重复执行”。

**大白话：**
消息队列不是说“绝对只送一次”，更像快递员有时怕你没签收，会多敲两次门。幂等就是告诉系统：就算同一份快递来了两次，也只能签收一次，不能领两份奖品。

#### 4.4 为什么这里选择 `paymentId` 做幂等标识？
**技术细节：**
因为 `paymentId` 是支付记录表的主键，天然唯一，而且一笔支付对应一次发券。相比只用 `userId`，`paymentId` 粒度更准确：同一个用户可以有多次充值，如果只按 `userId` 去重，会误伤正常业务。所以这里做的是“支付维度幂等”，而不是“用户维度幂等”。

**大白话：**
一个用户可以充值很多次，每次都应该拿到对应权益，所以不能拿“人”去去重，只能拿“这一次付款单号”去去重。

#### 4.5 这个模块后续还能怎么优化？
**技术细节：**
1. 给 `t_coupon.coupon_code` 增加数据库唯一索引，做最终兜底；
2. 将消费异常重试策略和死信队列策略做成可配置；
3. 将 `secret-key` 从配置文件迁移到环境变量；
4. 后续把加积分、发通知等动作也拆成独立消费者；
5. 如果消息体继续扩展，可以增加版本号字段，方便兼容不同消费者。

**大白话：**
现在已经够用了，但如果以后要上生产，还可以继续加“第二道保险”和“第三道保险”。比如数据库唯一索引就是最后一道闸门，哪怕代码哪里漏了，也不让重复发券真的落库。
