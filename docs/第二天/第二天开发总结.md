# 第二天开发总结：微信公众号接入与持久层搭建

## 面试官场景模拟 Q&A

### 模块一：微信公众号服务器验证与代码重构（第一件事）

#### 1. 你是如何设计这个模块的？
**👨‍💻 技术细节：**
在架构设计上，我遵循了**DRY（Don't Repeat Yourself）原则**和**单一职责原则（SRP）**。起初尝试手动实现 SHA-1 签名校验，但发现这会让基础加密逻辑污染业务控制层。随后我引入了业界成熟的开源库 `weixin-java-mp` (WxJava SDK)。
设计上，我摒弃了传统的 `static` 工具类写法，转而拥抱 Spring Boot 的 **IoC（控制反转）机制**。
*   **配置层 (`WxMpConfig.java`)**：创建了一个 `@Configuration` 配置类，通过 `@Value` 注解将 `appId`、`secret` 和 `token` 从 `application.properties` 中动态注入，并使用 `@Bean` 向 Spring 容器注册了一个单例的 `WxMpService`。
*   **控制层 (`WxResponse.java`)**：负责处理微信服务器发来的验证请求。仅需通过 `@Autowired` 注入 `WxMpService` 即可完成鉴权流转。

**🗣️ 大白话：**
一开始我想自己手写加密算法对暗号，后来觉得没必要重复造轮子，就用了一个大神写好的工具包（WxJava）。
设计的时候，我没有把账号密码写死在代码里，而是专门建了个“组装厂”（配置类 `WxMpConfig`），在项目启动时去配置文件里拿密码，组装成一个“微信工具箱”（Bean），放进 Spring 的公共仓库里。当前台（Controller `WxResponse`）来客人的时候，直接从仓库拿工具箱按一下“验伪按钮”就行了，分工明确，以后要改密码也不用重新打包代码。

#### 2. 你是如何实现这个模块的？
**👨‍💻 技术细节：**
1.  **依赖引入**：在 `pom.xml` 中添加 `weixin-java-mp` 依赖。
2.  **配置类实现 (`WxMpConfig.java`)**：实例化 `WxMpDefaultConfigImpl`，将 `@Value` 注入的 `appId` 等配置 set 进去。然后将其绑定到 `WxMpServiceImpl` 实例上，最后通过 `@Bean` 注解返回给 Spring 容器管理。
3.  **Controller 改造 (`WxResponse.java`)**：定义 `@GetMapping("/")` 接收微信服务器发来的 `signature`、`timestamp`、`nonce`、`echostr` 四个参数。通过调用 `wxMpService.checkSignature(timestamp, nonce, signature)` 进行校验。若返回 `true`，则直接通过 Spring MVC 的 `@RestController` 特性将 `echostr` 原样返回字符串，完成服务器归属权认证。

**🗣️ 大白话：**
先导包，然后写个配置类读取账号密码。接着在 Controller 里写个 GET 接口。当微信把时间戳、随机数和加密串发过来时，我直接调工具包里的 `checkSignature` 方法算一下。如果全对上了，我就把微信发过来的随机字符串（echostr）原封不动地还给它，微信就知道咱们是“自己人”了。

#### 3. 你是如何测试这个模块的？
**👨‍💻 技术细节：**
测试此模块的核心是解决公网回调问题。我使用了内网穿透工具，将本地的 `127.0.0.1:8081` 映射到公网 HTTPS 域名。然后在微信公众平台后台填写完整的公网 URL（精确到配置的路径）和 Token，点击提交。通过观察控制台日志未报错且微信后台提示“配置成功”，完成端到端集成测试。

**🗣️ 大白话：**
因为微信的服务器在公网上，它访问不到我个人的电脑，所以我用了一个“内网穿透”软件，给了我电脑一个公网能访问的网址。我去微信后台填好这个网址和口令，点一下提交。只要提示成功，并且我电脑控制台没报错，就说明测试通过了。

#### 4. 这个项目的八股文（底层原理拓展）
**👨‍💻 技术细节：**
*   **为什么弃用静态工具类（Static Util）改用 Spring Bean？** 静态类在 JVM 类加载时初始化，无法享受 Spring IoC 容器提供的依赖注入、动态刷新配置、AOP 代理等高级特性，是 Spring 框架中的反模式。
*   **微信鉴权为什么使用 SHA-1 且必须字典序排序？** SHA-1 是单向散列算法，用于防止数据被篡改。必须字典序排序是因为哈希算法具有“雪崩效应”，哪怕拼接顺序错了一个字母，生成的哈希值也会完全不同。保证双方拼接规则绝对一致，是鉴权成功的前提。
*   **`@Configuration` 和 `@Component` 有什么区别？** `@Configuration` 会被 CGLIB 进行代理，保证在其中调用 `@Bean` 方法时，如果容器中已经存在该 Bean，则直接返回容器中的实例，保证单例语义。

**🗣️ 大白话：**
*   **为啥不用 static**：用 static 就等于把东西锁死了，Spring 这个大管家管不到它。改成 Bean 就像是交给了大管家统筹安排，想什么时候换配置、加功能都非常灵活。
*   **加密和排序的意义**：排序是为了让咱们跟微信保持默契，大家按同一个顺序拼字符串。SHA-1 就像个“粉碎机”，不管多长的字放进去，出来的都是固定长度的乱码，别人截获了乱码也猜不出咱们原来的暗号是什么。
*   **@Configuration 的特殊之处**：它就像是个带锁的工厂，不管你找它要多少次“微信工具箱”，它都会把第一次做好的那个给你（单例），不会重复制造，省资源。

### 模块二：数据持久层搭建与用户实体映射（第二件事）

#### 1. 你是如何设计这个模块的？
**👨‍💻 技术细节：**
在持久层选型上，我选用了基于 MyBatis 增强的 **MyBatis-Plus (MP)**。设计的核心思想是“约定大于配置”与“贫血领域模型”。
我将实体类（`TUser`）与数据库表进行强映射。利用 `@TableName` 和 `@TableId` 精确指定表名和主键生成策略，并使用 Lombok 的 `@Data` 和 `@Builder` 减少样板代码。在 Mapper 层（`TUserMapper`），直接继承 `BaseMapper<TUser>`，这样设计将基础的单表 CRUD（增删改查）操作下沉到框架层自动处理，极大减少了 XML 文件的冗余编写，使得开发者能将精力集中在复杂的联表或业务 SQL 上。

**🗣️ 大白话：**
连数据库我选了 MyBatis-Plus 这个框架。为啥选它？因为它像个全自动洗衣机，最基本的增删改查它都帮你内置好了。设计上，我建了一个 Java 类（`TUser`），里面的属性跟数据库表 `t_user` 的列一一对应，打上几个注解告诉框架“这个类对应那张表”。然后建个接口（`TUserMapper`）继承框架给的模板。剩下的脏活累活全交给框架，不用我再手写一堆重复的 SQL 语句了。

#### 2. 你是如何实现这个模块的？
**👨‍💻 技术细节：**
1.  **环境与配置**：在 `pom.xml` 中引入 `mysql-connector-j` 和 `mybatis-plus-boot-starter`。在 `application.properties` 配置了 MySQL 连接信息及 MP 的控制台 SQL 打印。在 Spring Boot 启动类 `EasyevalApplication` 上添加 `@MapperScan("com.dong.easyeval.mapper")` 注解，统一扫描 Mapper 接口。
2.  **实体类实现 (`TUser.java`)**：创建 `TUser` 类，实现 `Serializable` 接口。使用 `@TableName("t_user")` 映射表名。使用 `@TableId(value = "user_id", type = IdType.AUTO)` 指定主键为自增类型。类上打上 `@Data`、`@Builder`、`@NoArgsConstructor`、`@AllArgsConstructor` 组合注解。
3.  **Mapper 接口实现 (`TUserMapper.java`)**：创建接口继承 `BaseMapper<TUser>`。虽然由于启动类已经有了 `@MapperScan`，此处的 `@Mapper` 注解是可选的，但我依然保留了它，以增强代码的语义可读性。

**🗣️ 大白话：**
先导包，写好数据库的账号密码配置。然后在项目的总入口（启动类）上加了个 `@MapperScan`，告诉程序去哪里找咱们写好的数据库接口。
接着建 `TUser` 实体类，用了 Lombok 插件，连 get/set 和构造方法都省了，直接用注解标记好哪个是表名、哪个是自增主键。
最后建个 `TUserMapper` 接口，直接继承框架给的模板，代码一行没写，就能直接操作数据库了。

#### 3. 你是如何测试这个模块的？
**👨‍💻 技术细节：**
可以通过编写基于 `@SpringBootTest` 的单元测试。使用 `@Autowired` 注入 `TUserMapper`，利用 `@Builder` 模式链式构造一个 `TUser` 测试对象。调用 `tUserMapper.insert(user)` 方法，断言返回值是否为 1。同时开启了 MP 的 SQL 日志打印，观察控制台生成的 `INSERT INTO t_user...` 语句是否符合预期，并到 MySQL 客户端验证数据是否落盘。

**🗣️ 大白话：**
测试很简单，写个测试用例。用代码自动生成一个假的用户信息，调一下 `insert` 存入数据库的方法。如果控制台打印出了正确的 SQL 语句，而且我去 Navicat（数据库可视化软件）里一刷新，真看到了这条数据，这模块就算跑通了。

#### 4. 这个项目的八股文（底层原理拓展）
**👨‍💻 技术细节：**
*   **关于 `@MapperScan` 和 `@Repository` 的区别？**
    *   `@MapperScan`：作用于启动类或配置类，指示 MyBatis 框架去哪个包下扫描接口，并通过 JDK 动态代理为这些接口生成实现类注册到 Spring 容器中。这是核心配置。
    *   `@Repository`：是 Spring 原生注解，本意是标记持久层的实现类并进行异常转换。对于 MyBatis 的 Mapper 接口来说，它并没有实际作用，仅仅是用来消除 IDEA 中 `@Autowired` 注入时产生的“找不到 Bean”的红线警告。
*   **主键为什么使用 `Long` 而不是 `Integer`？** MySQL 中的 `BIGINT` 对应 Java 的 `Long`。如果是互联网项目，用户量一旦突破 `Integer.MAX_VALUE`（约 21 亿），使用 Integer 就会导致溢出异常。
*   **Lombok 的 `@Builder` 是什么？** 实现了设计模式中的**建造者模式（Builder Pattern）**。解决了当类属性过多时，构造方法传参容易错位、连续调用 `set` 方法代码冗长的问题，提供了优雅的链式调用。

**🗣️ 大白话：**
*   **关于两个注解**：`@MapperScan` 就像是给大管家塞了一张“外包人员名单”，大管家照着名单把人全招进来，这个必须有。而 `@Repository` 就像是给这些人发个闲职胸牌，贴在接口上其实没啥大用，唯一的用处就是骗过 IDEA 这个“门卫”，让它别在代码里画报错的红线。
*   **为啥 ID 用 Long**：防患于未然。万一以后系统火了，数据超过 21 亿条，用 Integer 数据库就直接崩了，用 Long 稳如老狗。
*   **@Builder 是啥**：就像配电脑的流水线。你想要什么配置直接点（链式调用），`builder().给个CPU().给个内存().build()`，代码写起来特别顺手，看着也高级。

### 模块三：MyBatis-Plus 代码生成器配置与踩坑（第三件事）

#### 1. 你是如何设计这个模块的？
**👨‍💻 技术细节：**
为了提升开发效率并保证代码结构的统一性，我引入了 MyBatis-Plus 的 `FastAutoGenerator`（快速代码生成器）。
设计上，我将其作为一个独立的测试类工具（`CodeGenerator.java`），而非核心业务代码。通过链式编程（Builder Pattern）配置了四大策略：
1.  **全局配置（GlobalConfig）**：指定作者和绝对输出路径，禁止生成后自动打开目录。
2.  **包配置（PackageConfig）**：统一父包名 `com.dong.easyeval`，并将 `Mapper.xml` 剥离到 `resources/mapper` 目录下，符合 Maven 标准目录结构。
3.  **策略配置（StrategyConfig）**：精准指定需要生成的表名。开启 Lombok 实体模型，开启 Controller 的 `@RestController` 风格，以及 Mapper 接口的 `@Mapper` 注解。
4.  **模板引擎（TemplateEngine）**：底层依赖 `VelocityTemplateEngine` 渲染代码。

**🗣️ 大白话：**
因为数据库里有七八张表，要是每张表我都去手写 Entity、Mapper、Service、Controller，那太折磨人了。所以我搞了个“全自动代码打印机”（代码生成器）。
我给打印机设定了几个规则：生成的代码放哪、包名是什么、用不用 Lombok 偷懒、要不要给接口打上特定的标签。设定好之后，只要输入表名，一键运行，所有基础代码瞬间生成，这叫“磨刀不误砍柴工”。

#### 2. 你是如何实现这个模块的？
**👨‍💻 技术细节：**
1.  **引入依赖**：在 `pom.xml` 中加入 `mybatis-plus-generator` 和 `velocity-engine-core` 模板引擎依赖。
2.  **编写脚本**：在 `src/test/java` 下创建 `CodeGenerator` 包含 `main` 方法。配置好 JDBC URL、用户名和密码。
3.  **处理表前缀陷阱**：在生成过程中，我发现默认使用 `.addTablePrefix("t_")` 会导致生成的实体类丢失 `T` 前缀（例如 `t_user` 变成 `User`），这与我原有的架构设计（期望为 `TUser`）不符。我果断注释掉了 `.addTablePrefix("t_")`，并清理了错误的旧代码，重新生成了包含前缀的规范代码。

**🗣️ 大白话：**
先下好生成器需要的插件包。写个包含 `main` 方法的脚本，把数据库账号密码填进去。
中间踩了个坑：本来我想让生成器聪明点，结果它自作主张把表名里的 `t_` 给去掉了，导致生成的类名叫 `User`，而我想要的是 `TUser`。我马上反应过来，把那个“去前缀”的配置给注释掉，把生成错的垃圾文件全删了，重新跑了一遍，这才得到了我想要的整整齐齐的目录。

#### 3. 你是如何测试这个模块的？
**👨‍💻 技术细节：**
代码生成器本身的测试就是运行 `main` 方法。
生成完成后，我通过两步验证：
1.  **结构验证**：检查项目目录树，确认 Controller、Service、Mapper、Entity 均已生成，且 `Mapper.xml` 正确落入 `resources` 目录。
2.  **连通性验证**：修改之前编写的 `TuserMapperTest` 单元测试，将手动编写的 `TUser` 替换为自动生成的 `TUser`。由于新生成的实体类未开启 `@Builder`，我将测试代码重构为标准的 `new TUser()` 和 `set` 方法。同时引入了 `System.currentTimeMillis()` 拼接 `wechatUserId`，以规避数据库唯一索引造成的重复插入报错。测试最终完美通过（Green Bar）。

**🗣️ 大白话：**
点下绿色箭头运行脚本，看左边目录有没有唰唰唰多出一堆代码。
然后去跑一下咱们之前的数据库测试类。因为新生成的代码没开那种花里胡哨的链式调用，我就老老实实改成用 `set` 方法赋值。我还很机智地在测试的微信 ID 后面加了个时间戳，这样就算我点一百次测试，也不会因为数据库里已经有这个微信号而报错。最后测试绿灯亮起，搞定！

#### 4. 这个项目的八股文（底层原理拓展）
**👨‍💻 技术细节：**
*   **代码生成器的底层原理是什么？** 基于**模板引擎（Template Engine）**技术。MyBatis-Plus 内部读取数据库的元数据（Database MetaData，如表名、列名、字段类型），将其映射为 Java 上下文对象（如 `TableInfo`）。然后结合内置的 Velocity 或 Freemarker 模板（`.vm` 或 `.ftl` 文件），将上下文变量填充到模板占位符中，最终通过 I/O 流输出为 `.java` 源码文件。
*   **为什么要把 `CodeGenerator` 放在 `src/test/java` 下，而不是 `src/main/java`？** 因为代码生成器仅在开发阶段作为脚手架工具使用，属于**开发期依赖**。放在 `test` 目录下可以保证在执行 `mvn package` 打包生产环境的 jar 包时，不会将生成器代码和相关的模板引擎依赖（如 Velocity）打包进去，从而减小部署包的体积，提高安全性。

**🗣️ 大白话：**
*   **代码生成器是怎么干活的**：它其实就像个“填空题机器”。框架底层写好了很多带空格的文章（模板），比如 `public class ${className} { }`。然后它跑去数据库查一下表名，发现叫 `TUser`，就把 `TUser` 填进空格里，最后保存成一个真正的 Java 文件。
*   **为啥要把这玩意放进 test 文件夹**：因为这东西就是个盖房子用的“脚手架”。房子（项目）盖好了，你要把它卖给客户（上线打包），总不能把脚手架也一起打包卖过去吧？放在 test 里，打包的时候就会自动忽略它，既省空间又安全。

### 模块四：代码重构与 Lombok Builder 模式支持（第四件事）

#### 1. 你是如何设计这个模块的？
**👨‍💻 技术细节：**
在之前的生成代码中，由于 MyBatis-Plus FastAutoGenerator 默认的 `enableLombok()` 仅包含 `@Getter` 和 `@Setter` 注解，导致实体类不支持 `@Builder` 建造者模式。这破坏了原本在 `PayController` 和 `TuserMapperTest` 中使用的链式调用写法，降低了代码的可读性。
设计上，为了保持代码的优雅和一致性，我决定**推翻重来**。我开启了代码生成器的**文件覆盖模式（`enableFileOverride()`）**，并在生成后，通过自动化脚本或自定义模板的方式，将实体类的普通 Lombok 注解替换为包含 `@Data`、`@Builder`、`@NoArgsConstructor` 和 `@AllArgsConstructor` 的全套组合。

**🗣️ 大白话：**
一开始生成的代码只能规规矩矩地用 `set` 方法去挨个赋值，代码写起来又长又丑。我觉得这样不行，之前我们用 `builder()` 链式调用写得多爽啊！所以我直接把刚才生成的代码全删了，给生成器加了个“强制覆盖”的功能。因为原生的生成器有点笨，不自带 Builder，我就在它生成完之后，用脚本把那些不够高级的注解替换成了全套的 Lombok 注解，让所有实体类瞬间恢复了“高级感”。

#### 2. 你是如何实现这个模块的？
**👨‍💻 技术细节：**
1.  **清理历史包袱**：首先通过 PowerShell 脚本删除了之前生成的不带 Builder 的所有 `entity`、`mapper`、`service` 和 `controller` 文件。
2.  **升级生成器配置**：在 `CodeGenerator.java` 中，为 Entity、Controller、Mapper、Service 四个策略都添加了 `.enableFileOverride()` 允许覆盖文件。
3.  **自动化注入 Builder**：重新运行代码生成器后，编写 PowerShell 脚本遍历所有生成的 `.java` 实体类文件。使用正则表达式将 `import lombok.Getter;` 和 `import lombok.Setter;` 替换为全套的 `@Data` 和 `@Builder` 依赖，并将类上的 `@Getter` 和 `@Setter` 注解替换为 `@Data`、`@Builder`、`@AllArgsConstructor`、`@NoArgsConstructor`。

#### 4. 这个项目的八股文（底层原理拓展）
**👨‍💻 技术细节：**
*   **Lombok 的 `@Data` 和 `@Builder` 一起使用为什么需要加全参和无参构造（`@AllArgsConstructor`、`@NoArgsConstructor`）？**
    单独使用 `@Data` 时，Lombok 会生成一个默认的无参构造。但如果加上 `@Builder`，Lombok 会生成一个全参构造来供 Builder 内部类使用，这会导致默认的无参构造消失。而 MyBatis-Plus 和 Spring 在进行反射实例化（如查询数据库映射到实体类，或 JSON 反序列化）时，**强依赖无参构造函数**。因此，必须显式加上 `@NoArgsConstructor` 和 `@AllArgsConstructor`，否则在运行时会抛出 `MybatisPlusException: No constructor found` 或 `HttpMessageNotReadableException`。

**🗣️ 大白话：**
*   **为啥用了 Builder 还要加一堆构造方法的注解？**
    你可以这么理解：`@Data` 本来会贴心地给你准备一个“空房间”（无参构造），让框架能随时进去放数据。但是你一旦用了 `@Builder`，它为了自己方便，直接把这个房间塞满了家具（全参构造），原来的空房间没了！MyBatis 框架想去数据库拿数据塞进类里的时候，发现没有空房间可以进，就会当场崩溃报错。所以，我们必须手动加上那两个注解，告诉 Lombok：“兄弟，全参的房间你要留着，但也必须给我留一个空房间给框架用！”

### 模块五：支付流程集成与雪花算法优化（第五件事）

#### 1. 你是如何设计这个模块的？
**👨‍💻 技术细节：**
本模块旨在实现一个后台管理员为用户手动充值并发放卡券的接口。
在架构设计上，我遵循了**高内聚低耦合**的原则：
1. **数据传输对象（DTO）分离**：创建了 `PayRequest` 类专门用于接收前端 JSON 参数，使用 `@JsonProperty` 注解将前端的下划线命名（如 `wechat_id`）映射为 Java 的驼峰命名（`wechatId`），隔离了前后端命名规范的差异。
2. **唯一标识生成**：为了保证每次充值生成的兑换码具有全局唯一性且趋势递增，我引入了 **Twitter 的 Snowflake（雪花算法）**（`SnowflakeIdGenerator`）。并通过 `@Configuration` 将其注册为单例 Bean（`SnowflakeIdGeneratorConfig`），由于当前是单机部署，我在构造器中将其 `datacenterId` 和 `machineId` 默认硬编码为 `1L`，去除了对配置文件的强依赖。
3. **业务编排**：在 `PayController` 中，使用统一的 `ApiResponse` 包装返回结果。充值逻辑被拆分为三个内聚的步骤：校验参数与查询用户 -> 插入支付流水记录（`TPaymentHistory`） -> 生成并下发兑换券（`TCoupon`），全程使用 Lombok 的 `@Builder` 模式构建实体对象。

**🗣️ 大白话：**
这个模块是给管理员用的充值页面。设计的时候我分了三步走：

最后，在控制器（Controller）里，我像流水线一样安排了工作：先查查有没有这个用户 -> 把他交的钱记到账本上 -> 最后调动雪花算法给他发一张有效期半年的卡券。

#### 4. 这个项目的八股文（底层原理拓展）
**👨‍💻 技术细节：**
* **为什么要用雪花算法（Snowflake）而不是 UUID 或数据库自增 ID？**
  * **数据库自增 ID**：在分布式系统、分库分表场景下会产生 ID 冲突，且容易被竞争对手爬取暴露业务体量（如订单量）。
  * **UUID**：虽然能保证全局唯一，但它是 32 位无序的字符串。作为 MySQL 的主键（InnoDB 聚簇索引的 B+ 树）插入时，会导致频繁的**页分裂（Page Split）**和数据移动，严重降低写入性能和空间利用率。
  * **雪花算法**：生成的 64 位 Long 型整数，既保证了分布式环境下的**全局唯一性**，又因为包含时间戳前缀，保证了主键是**趋势递增**的。完美契合 B+ 树的顺序插入特性，极大地提高了数据库性能。
* **`@JsonProperty` 的底层原理是什么？如果在 Spring Boot 中不用它还有什么办法解决下划线转驼峰？**
  * 它是 Jackson 库的注解，在反序列化时通过反射机制将 JSON 键匹配到实体字段。如果不加注解，可以在 `application.properties` 中全局配置 `spring.jackson.property-naming-strategy=SNAKE_CASE` 来自动实现下划线与驼峰的全局转换。
* **JS 中的 `==` 和 `===` 有什么区别？**
  * `==`（松散相等）：比较前会进行**隐式类型转换**。比如 `200 == "200"` 结果为 `true`。
  * `===`（严格相等）：不进行类型转换，要求**值和数据类型必须完全一致**。比如 `200 === "200"` 结果为 `false`。在现代前端开发中，推荐永远使用 `===` 以避免不可预期的隐式转换 Bug。

**🗣️ 大白话：**
* **为啥发券的码要用雪花算法**：用数据库自己往上加的 ID 不行，万一以后系统做大了分了好几个库，ID 就撞车了，而且别人下个单就知道你今天卖了多少单。用那种长串的字母 UUID 也不行，因为数据库存数据就像排队，UUID 没按顺序来，到处插队，把数据库搞得手忙脚乱，性能极差。雪花算法生成的数字不仅绝不重复，而且是越变越大的，数据库存起来特别舒服，生成速度也快得飞起！
* **`@JsonProperty` 是咋工作的**：它就像个翻译官贴纸。后端收到一串别人发来的数据包，它一看这贴纸，就知道“噢，这个带下划线的名字其实就是指我这个变量”。如果不想每个变量都贴贴纸，也可以直接去大管家（配置文件）那里定个规矩，让它以后自动把下划线都当成驼峰。
* **三个等号是啥**：在 JS 里，两个等号 `==` 比较随便，数字 1 和字符串 "1" 它觉得是一家人。三个等号 `===` 就很严格，不但值要一样，血统（数据类型）也得一模一样才行。为了代码不出诡异的 bug，老司机都用三个等号。
