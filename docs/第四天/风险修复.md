# 第四天：风险修复

## 一、今天修了什么

今天这次不是继续堆新功能，而是把项目里已经暴露出来的几个“高风险点”真正收口，重点修了 5 类问题：

1. **卡券兑换接口的数据一致性问题**
2. **支付记录与 RocketMQ 发送之间的链路一致性问题**
3. **RocketMQ 消费端重复发券的幂等问题**

## 二、风险 1：卡券兑换接口的一致性问题

### 1. 问题现象

原来的 `CouponController.exchange()` 有两个明显风险：

- 当用户第一次兑换卡券、`t_correction_count` 里还没有额度记录时，代码会先 `build()` 一个空对象，然后直接执行：

```java
tCorrectionCount.setTotalCount(tCorrectionCount.getTotalCount() + 20);
```

- 这里 `getTotalCount()` 会直接空指针。
- 即使不空指针，这个新对象也没有补 `userId`、`usedCount`、`expirationDate` 这些关键字段，落库的数据也不完整。

另一个更大的问题是：

- 代码先把券状态改成 `USED`
- 再去给用户增加批改次数

如果第二步失败，就会出现：

- **券被消耗了**
- **额度却没加上**

这就是典型的数据不一致。

### 2. 设计思路

这里我的设计目标很明确：**卡券状态更新和额度发放必须视为一个原子操作**。

也就是说：

- 要么两步都成功
- 要么两步都失败并回滚

所以这里采用的是：

- `@Transactional(rollbackFor = Exception.class)` 把兑换操作包成一个本地事务
- 对“首次兑换没有额度记录”的用户，初始化一条完整的 `TCorrectionCount`
- 对已有额度记录的用户，安全地把 `total_count` 增加 20
- 如果中间任一更新失败，整个事务回滚

### 3. 实现细节

修复位置：

- `src/main/java/com/dong/easyeval/controller/CouponController.java`

这次具体做了这些改动：

#### （1）给兑换接口加事务

```java
@PostMapping("exchange")
@Transactional(rollbackFor = Exception.class)
public ApiResponse exchange(@RequestBody ExchangeRequest request)
```

这样券状态更新和额度更新就在同一个事务里了。

#### （2）先校验参数，再把 `userId` 转成 `Long`

原来直接拿字符串去查 `user_id`，现在先做：

```java
Long userId = Long.valueOf(request.getUserId());
```

如果格式不对，直接返回错误，不再继续执行。

#### （3）首次兑换时初始化完整额度记录

如果查不到 `t_correction_count`：

```java
TCorrectionCount.builder()
    .userId(userId)
    .totalCount(20)
    .usedCount(0)
    .expirationDate(LocalDate.now().plusMonths(6))
    .build();
```

这里补齐了：

- `userId`
- `totalCount`
- `usedCount`
- `expirationDate`

#### （4）已有额度时做安全累加

为了避免空值问题，这里用了：

```java
int currentTotalCount = Objects.requireNonNullElse(tCorrectionCount.getTotalCount(), 0);
int currentUsedCount = Objects.requireNonNullElse(tCorrectionCount.getUsedCount(), 0);
```

然后再更新：

```java
tCorrectionCount.setTotalCount(currentTotalCount + 20);
tCorrectionCount.setUsedCount(currentUsedCount);
```

#### （5）更新失败时显式回滚

我加了一个统一的 `rollbackWithError()`：

```java
TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
```

这样当券状态更新失败、或者额度保存失败时，都会把事务标记为回滚。

### 4. 大白话解释

你可以把“兑换卡券”理解成收银台里的两个动作：

1. 先把卡券撕掉，表示已使用  
2. 再给用户账户里充值 20 次批改额度

以前的问题是：  
万一第 2 步失败了，第 1 步已经做了，用户就会觉得“券没了，但额度也没到账”。

现在加了事务之后，这两个动作就像绑在一起了：

- 成功就一起成功
- 失败就一起撤回

## 三、风险 2：支付记录和 MQ 发送之间的链路一致性问题

### 1. 问题现象

原来的 `PayController.pay()` 先做：

1. 保存 `t_payment_history`
2. 再发送 RocketMQ 消息

这样会有一个非常现实的问题：

- 数据库支付记录已经写进去了
- 但是 MQ 如果发送失败，接口却返回错误

结果就会变成：

- 用户/管理员看到“充值失败”
- 数据库里却已经有一条支付记录

这会让系统状态和用户感知不一致。

### 2. 设计思路

这一版修复的目标是：

- **至少先解决“消息立即发送失败时，支付记录不能留脏数据”**

所以我采用的是：

- 先检查 RocketMQ 是否启用、Producer 是否初始化完成
- 再写支付记录
- 写完后发送 MQ
- 如果序列化失败或发送失败，当前事务回滚

> 说明：  
> 这次修的是“本地事务内能控制住的即时失败”。  
> 如果后面要继续升级成更强的一致性方案，可以再做 Outbox / 事务消息，这是下一阶段优化，不是这次修复的主目标。

### 3. 实现细节

修复位置：

- `src/main/java/com/dong/easyeval/controller/PayController.java`

#### （1）给支付接口加事务

```java
@PostMapping("/pay")
@Transactional(rollbackFor = Exception.class)
public ApiResponse pay(@RequestBody PayRequest payRequest)
```

#### （2）把 MQ 前置检查放到支付记录保存之前

现在先判断：

```java
if (!rocketMq5Properties.isEnabled()) { ... }
if (Objects.isNull(rocketMqProducer)) { ... }
```

这样如果 MQ 根本没配好，就不会先落库再报错。

#### （3）给支付状态兜底

前端实际没有传 `payment_status`，所以这里做了默认值：

```java
String paymentStatus = StringUtils.defaultIfBlank(payRequest.getPaymentStatus(), "SUCCESS");
```

避免写出空状态支付记录，也避免消费者拿到空状态后误处理。

#### （4）保存支付记录时检查返回值

```java
if (!paymentHistoryService.save(paymentHistory)) {
    return rollbackWithError("支付记录保存失败");
}
```

#### （5）消息序列化或发送失败时回滚

这次发送的是结构化消息：

```java
PayFinishedMessage.builder()
    .paymentId(paymentHistory.getPaymentId())
    .userId(user.getUserId())
    .paymentStatus(paymentStatus)
    .build();
```

如果：

- JSON 序列化失败
- RocketMQ 发送失败

都会走：

```java
rollbackWithError(...)
```

从而撤回这次支付记录。

### 4. 大白话解释

以前像这样：

- 先登记“这人充了值”
- 再发消息通知后面的系统去送券

如果通知没发出去，前台还显示失败，但后台已经偷偷登记成功了。

现在的做法就是：

- 只要后面的通知没发出去，这次充值记录也一起撤回

这样至少不会出现“前台说失败、后台却记成功”的脏状态。

## 四、风险 3：RocketMQ 消费端重复发券的幂等问题

### 1. 问题现象

原来的 `MessageConsumer` 做的是：

1. 先查数据库里有没有这张券
2. 没有再插入

这个逻辑看起来没问题，但在并发场景下有漏洞：

- 两条相同消息几乎同时到达
- 两边都先查到“没有”
- 然后两边都去插入

这就是典型的“先查后写”竞态条件。

### 2. 设计思路

这里真正可靠的做法不是靠 Java 代码里“先查一下”，而是：

- **把幂等兜底交给数据库唯一约束**

因为 `t_coupon.coupon_code` 本身已经有唯一索引，所以最稳的方案是：

1. 直接按 `paymentId` 生成固定券码  
2. 直接插入  
3. 如果数据库抛唯一键冲突，说明这张券已经发过了，直接当成功处理

这就把并发情况下的最终裁决权交给数据库了。

### 3. 实现细节

修复位置：

- `src/main/java/com/dong/easyeval/consumer/MessageConsumer.java`

#### （1）消息体里增加 `paymentId`

支付端现在发出去的消息包含：

- `paymentId`
- `userId`
- `paymentStatus`

消费者拿到后用：

```java
String couponCode = "PAY-" + payFinishedMessage.getPaymentId();
```

这意味着：

- 同一笔支付
- 永远只会映射到同一张券

#### （2）只处理成功支付消息

```java
if (!"SUCCESS".equalsIgnoreCase(payFinishedMessage.getPaymentStatus())) {
    return ConsumeResult.SUCCESS;
}
```

这样可以避免失败支付也发券。

#### （3）直接插入，捕获唯一键异常

```java
try {
    couponService.save(...)
} catch (DuplicateKeyException e) {
    log.info("Skip duplicate coupon issue...");
    return ConsumeResult.SUCCESS;
}
```

这一步是这次幂等修复的关键。

#### （4）增加消费日志

现在消费者会打印 4 类关键日志：

- 收到消息：`Receive pay finished message`
- 命中幂等：`Skip duplicate coupon issue`
- 发券成功：`Coupon issued successfully`
- 消费失败：`Consume pay finished message failed`

### 4. 大白话解释

MQ 有一个很常见的特点：  
**同一条消息，系统要默认它有可能被重复投递。**

所以我们不能想当然地认为：

- “消息只会来一次”

而应该按：

- “就算来两次，我也只能发一张券”

来设计。

这次我们用的办法就是：

- 同一笔支付固定生成同一个 `couponCode`
- 数据库自己保证这个 `couponCode` 不能插入两次

这样即使 MQ 重复投递，也不会重复发券。

## 五、风险 4：管理员支付接口 IP 过滤器不稳

### 1. 问题现象

原来的 `IpFilter` 有几个问题：

- IP 写死在代码里：`192.168.20.102`
- `X-Forwarded-For` 不拆分，多个代理 IP 时会整个字符串比较
- 拒绝访问时没有明确设置 403 状态码

这样一旦：

- 你换了公网 IP
- 前面多了一层 Nginx / 内网穿透
- 代理头里出现多个 IP

过滤器就会误判。

### 2. 设计思路

这个模块的设计目标是：

- **把允许访问的 IP 放到配置里，不写死在代码里**
- **兼容代理头，优先取真实客户端 IP**
- **拒绝时返回标准 403**

### 3. 实现细节

修复位置：

- `src/main/java/com/dong/easyeval/fillter/IpFilter.java`
- `src/main/java/com/dong/easyeval/config/FilterConfig.java`
- `src/main/java/com/dong/easyeval/config/AdminSecurityProperties.java`
- `src/main/resources/application.properties`

#### （1）把白名单提到配置

新增：

```properties
admin.security.allowed-ips=127.0.0.1,::1,192.168.20.102,116.27.5.255
```

#### （2）用配置类承接白名单

新增 `AdminSecurityProperties`：

```java
@ConfigurationProperties(prefix = "admin.security")
```

这样以后改 IP 不需要改 Java 代码。

#### （3）过滤器支持多 IP 代理头

现在如果 `X-Forwarded-For` 是：

```text
116.27.5.255, 10.0.0.2
```

会先取第一个：

```java
clientIp = clientIp.split(",")[0].trim();
```

#### （4）拒绝时返回标准 403

```java
httpServletResponse.setStatus(HttpServletResponse.SC_FORBIDDEN);
```

### 4. 大白话解释

原来这个过滤器像门卫只认识一张写死的身份证号，其他一律不让进。  
但真实线上环境里，经常还会经过网关、代理、穿透工具，IP 不一定只有一个。

现在等于把门卫规则改成了：

- 白名单放在配置里，想换直接改配置
- 遇到代理头，优先识别最前面的真实来源 IP
- 不允许就标准 403 拒绝

这样更像真正可上线的实现。

## 六、风险 5：依赖和测试环境不稳定

### 1. 问题现象

原来的 `pom.xml` 混用了：

- Spring Boot 3.3.13
- `spring-boot-starter-web 2.7.17`
- `javax.servlet-api`
- `servlet-api 2.5`

这其实是两套时代的 Servlet 体系：

- Boot 3 用的是 `jakarta.servlet`
- Boot 2 / 老项目常见的是 `javax.servlet`

虽然短期可能还能编译，但长期很容易出类路径冲突。

另外测试环境也有两个问题：

- SpringBootTest 会把真实 RocketMQ 配置带进来
- 微信菜单发布测试会访问真实微信接口

这会让 `mvn test` 不稳定。

### 2. 设计思路

目标是：

- 统一依赖到 Boot 3 的技术栈
- 让自动化测试默认跑在“不会连外部 MQ、不会误调微信”的模式下

### 3. 实现细节

#### （1）清理旧 Servlet 依赖

修复位置：

- `pom.xml`

移除了：

- `spring-boot-starter-web` 上手工指定的 `2.7.17`
- `javax.servlet-api`
- `servlet-api 2.5`

现在统一交给 Spring Boot 3 的依赖管理。

#### （2）给测试环境单独关掉 RocketMQ

新增：

- `src/test/resources/application-test.properties`

内容：

```properties
rocketmq5.enabled=false
admin.security.allowed-ips=127.0.0.1,::1
```

#### （3）给 Surefire 指定测试 profile 和 RocketMQ 日志目录

在 `pom.xml` 里新增：

- `spring.profiles.active=test`
- `rocketmq.client.logRoot=${project.build.directory}/rocketmq-logs`

这样测试时不会再去用默认用户目录写 RocketMQ 日志。

#### （4）把微信菜单测试改成手工集成测试

`WxMenuPublisherTest` 加了：

```java
@Disabled("Manual integration test against WeChat API")
```

因为它本质上不是单元测试，而是一个会打到微信公网的手工发布脚本。

#### （5）数据库测试加回滚

`TuserMapperTest` 现在加了：

- `@Transactional`
- `@Rollback`

这样测试不会污染本地数据库。

### 4. 大白话解释

这一步其实就是“把项目从能跑，修成能稳定测试”。

原来是：

- 依赖有点混
- 测试一跑就可能连真实 MQ
- 还有测试会直接调微信接口

现在改成：

- Boot 3 统一栈
- 测试默认走 test profile
- 外部依赖默认关掉或人工触发

这样以后你每次跑 `mvn test`，不会像开盲盒一样。

## 八、面试官视角：你是如何设计这个模块的？

### 面试题 1：你是如何设计“充值完成后发券”这个模块的？

#### 技术细节回答

我把“充值完成”和“发券”拆成了两个职责不同的模块：

1. **支付入口模块**
   - 负责参数校验
   - 查询用户
   - 保存支付记录
   - 发送 `PAY_FINISH` 事件到 RocketMQ

2. **发券消费模块**
   - 订阅 `PAY_FINISH`
   - 解析 `paymentId/userId/paymentStatus`
   - 只处理成功支付
   - 根据 `paymentId` 生成固定 `couponCode`
   - 插入优惠券

之所以这么拆，是因为“支付完成”是一个事件，而“发券”只是这个事件的后续动作之一。  
如果以后要扩展：

- 发积分
- 发站内通知
- 记运营日志

都可以继续订阅同一个支付完成事件，不需要再改支付入口本身。

#### 大白话解释

就是把“充值”当成一个上游动作，“送券”当成下游动作。  
充值成功后，不是自己当场把所有事都做完，而是先广播一句：

> 这笔支付完成了

后面谁需要响应这个事件，谁自己去处理。

## 九、面试官视角：你是如何实现这个模块的？

### 面试题 2：你是如何落地实现支付解耦和幂等消费的？

#### 技术细节回答

我这次使用的是：

- `RocketMQ 5.x Java SDK`
- `Producer.send(...)` 发送支付完成消息
- `MessageListener` 实现消费逻辑

核心实现点有 4 个：

1. **消息体结构化**
   - 不是只传 `userId`
   - 而是传 `paymentId + userId + paymentStatus`
   - 这样消费者可以做更准确的业务判断

2. **固定业务幂等键**
   - 券码直接设计成：`PAY-<paymentId>`
   - 这样一笔支付天然只对应一张券

3. **数据库唯一索引兜底**
   - `t_coupon.coupon_code` 已有唯一索引
   - 消费端直接插入
   - 捕获 `DuplicateKeyException` 后直接返回成功

4. **消费日志可观测**
   - 收消息、跳过重复、发券成功、消费失败，都有明确日志

#### 大白话解释

说白了就是：

- 发消息时，别只说“给这个人发券”
- 而要说清楚“这是第几笔支付、是谁、状态是什么”

这样消费者拿到消息后，才能准确知道：

- 该不该处理
- 会不会重复
- 重复了该怎么安全跳过

## 十、面试官视角：你是如何测试这个模块的？

### 面试题 3：你怎么证明这套风险修复真的有效？

#### 技术细节回答

我分了三层验证：

1. **编译层**
   - 先跑 `mvn -DskipTests compile`
   - 确保改动没有语法或依赖问题

2. **单元测试层**
   - `CouponControllerTest` 验证兑换分支逻辑

## 十一、这个模块相关的八股文

### 1. 本地事务和数据一致性

**技术点：**

- Spring 里的 `@Transactional`
- 默认传播行为是 `REQUIRED`
- 当前场景下适合把“改券状态 + 发放额度”包成一个本地事务

**大白话：**

就是把多个数据库动作绑在一起，别出现做了一半就停住的情况。

### 2. MQ 为什么一定要做幂等

**技术点：**

- RocketMQ 这类消息系统通常按“至少投递一次”设计
- 也就是说，消费者必须默认消息可能重复到达
- 所以业务侧要保证“重复消费不会产生重复结果”

**大白话：**

MQ 像快递员，系统默认它有可能多敲你一次门。  
所以你不能因为快递员多敲一次门，就多送一次券。

### 3. 为什么“先查后写”不是真正幂等

**技术点：**

- `select -> if null -> insert` 在并发下会有竞态条件
- 真正稳的是：
  - 唯一键
  - 乐观锁
  - 原子 upsert

这次项目里选的是：**唯一键 + 捕获重复键异常**

**大白话：**

你自己先看一眼“有没有”，再去写，别人也可能同时在做同样的事。  
最稳的是让数据库这个“裁判”来决定谁能写进去。

### 4. `X-Forwarded-For` 为什么不能整串拿来比

**技术点：**

- 经过代理后，`X-Forwarded-For` 可能是一个 IP 列表
- 格式通常是：

```text
客户端IP, 代理1IP, 代理2IP
```

- 真实客户端 IP 一般取第一个

**大白话：**

代理头里可能写的是一串人名，不是一个人名。  
你要找真正来访的人，就得拿第一个，而不是整串一起比。

### 5. Spring Boot 3 为什么不能再混 `javax.servlet`

**技术点：**

- Boot 3 迁到了 Jakarta EE
- Servlet API 包名从 `javax.servlet.*` 变成了 `jakarta.servlet.*`
- 如果项目里还混着老的 `javax.servlet` 依赖，很容易出现类不兼容或自动配置异常

**大白话：**

这就像框架已经升级到“新身份证体系”了，你项目里还夹着一套“旧身份证体系”，短期可能凑合，长期一定容易出错。

## 十三、OAuth `code` 重复使用导致“未知错误”的风险修复

### 1. 问题现象

在英语作文批改页面 `src/main/resources/static/eaasyeval.html` 中，用户第一次提交作文后可以正常拿到批改结果；但如果不重新从公众号菜单进入页面，而是在当前页面里只是切换年级、修改作文内容后再次点击“提交批改”，后端就会报：

```text
错误代码：40163
错误信息：oauth_code已使用
```

前端原来的表现是直接弹出：

```text
批改失败：未知错误
```

这对真实用户来说体验很差，因为用户根本不知道自己错在哪里，也不知道下一步该怎么做。

### 2. 根因分析

这个问题的根因不是作文批改逻辑本身，而是**微信 OAuth 授权码 `code` 的使用规则**：

- 公众号网页授权返回的 `code` **只能使用一次**
- 你的后端在 `EccController.submit()` 中，每次提交作文都会调用微信接口，把 `code` 再换一次 `openId`
- 第一次提交时，`code -> openId` 成功
- 第二次提交时，还是同一个页面、还是同一个 `code`，于是微信直接返回 `40163 code been used`

也就是说，当前问题的本质是：

> 页面复用了旧的 `code`，而后端又把这个 `code` 当成“每次都能重新兑换”的临时票据。

### 3. 这次是如何设计修复方案的

这次我没有只在前端弹一个更友好的提示，而是从**后端授权链路**上做了修复，因为真正的问题出在服务端对 OAuth `code` 的使用方式上。

我最终采用的是：

- 新增一个统一的授权解析服务 `WxOAuthCodeService`
- 第一次成功用 `code` 换出 `openId` 后，把映射关系短暂缓存到 Redis
- 同一个页面在短时间内再次提交时，优先从缓存里拿 `openId`
- 如果微信明确返回：
  - `40163`：`code` 已使用
  - `40029`：`code` 已失效
  就向业务层抛出**可读的业务异常信息**
- 前端再把这个业务异常展示成“请重新从公众号菜单进入页面”这种用户能理解的话

这个设计有两个好处：

1. **不让同一页面的短时间二次提交直接炸掉**
2. **即使真的过期了，也能给用户明确的操作指引，而不是“未知错误”**

### 4. 具体实现细节

#### 4.1 新增统一授权服务

我新增了：

- `src/main/java/com/dong/easyeval/service/WxOAuthCodeService.java`
- `src/main/java/com/dong/easyeval/common/WxOAuthCodeException.java`

`WxOAuthCodeService` 的职责是：

- 校验 `code` 是否为空
- 先查 Redis 缓存：`wx:oauth:code:<code>`
- 如果缓存有值，直接返回 `openId`
- 如果缓存没有值，再调用微信 OAuth 接口
- 首次成功时，把 `code -> openId` 缓存 5 分钟
- 如果微信返回 `40163` 或 `40029`，转换成友好的业务异常

这里的关键代码是：

- `src/main/java/com/dong/easyeval/service/WxOAuthCodeService.java`

其中用了 Redis 做短期缓存，缓存时间是 300 秒。这个时间足够覆盖用户在页面里连续提交、切换年级、微调作文内容等正常操作。

#### 4.2 接入到 3 条业务链路

这次不是只修了作文批改，而是把所有依赖微信 `code -> openId` 的接口一起统一接到了新服务上：

- 作文提交：`src/main/java/com/dong/easyeval/controller/EccController.java`
- 批改记录查询：`src/main/java/com/dong/easyeval/controller/EccController.java`
- 我的卡券：`src/main/java/com/dong/easyeval/controller/CouponController.java`

这样做的原因是：

- 如果只修 `submit`
- 但 `eval_record` 或 `mycoupon` 还保留旧逻辑
- 后面用户还是会在其他页面继续遇到同类问题

所以这次是按“统一授权入口”的思路做的，而不是打一块补丁。

#### 4.3 前端提示语优化

我同时优化了前端页面：

- `src/main/resources/static/eaasyeval.html`

新增了 `getFriendlyErrorMessage()`，现在页面会区分：

- 普通失败
- 微信授权重复使用
- 服务器返回非 JSON

当后端返回类似：

```text
当前页面授权已被使用，请返回公众号菜单重新进入后再提交
```

前端会继续加工成更符合用户操作习惯的话：

> 批改失败：当前页面授权已被使用，请返回公众号菜单重新进入当前页面。

这比“未知错误”更好，因为它至少告诉用户：

- 不是作文内容有问题
- 不是系统随机坏了
- 而是当前页面授权票据过期/已用掉了

### 6. 面试官如果问：你是如何设计这个模块的？

#### 技术版回答

我把“微信网页授权 `code` 解析”从 Controller 中抽离成了一个独立服务 `WxOAuthCodeService`。原因是多个接口都依赖同样的授权流程：作文批改提交、批改记录查询、我的卡券查询都需要 `code -> openId -> userId`。如果每个 Controller 都各自直接调用微信 SDK，一旦微信 OAuth 的错误处理策略发生变化，就需要在多个地方重复修改。

因此我做了两层设计：

1. **统一授权解析层**
   - 负责 `code` 判空
   - 负责 Redis 缓存
   - 负责调用微信 SDK
   - 负责把微信错误码转换为业务异常

2. **业务控制层**
   - 只关心拿到 `openId` 后继续查用户、查额度、查卡券、提交作文
   - 不直接感知微信错误码细节

这相当于把第三方授权协议的复杂性收敛到了一个服务里，让 Controller 保持“面向业务”的清晰职责。

#### 大白话回答

我把“找微信换身份信息”这件事，从每个接口里单独拿出来，做成了一个公共小助手。以后谁要 `openId`，都先找这个小助手拿。这样微信规则变了，我只改一个地方就行，不用满项目到处找。

### 7. 面试官如果问：你是如何实现这个模块的？

#### 技术版回答

实现上，核心是 **“先缓存，后兑换，失败兜底”**：

1. Controller 收到前端带来的 `code`
2. 调用 `WxOAuthCodeService.resolveOpenId(code)`
3. 服务先查 Redis 是否已有 `wx:oauth:code:<code>`
4. 如果有，直接返回缓存的 `openId`
5. 如果没有，再调用：
   - `wxMpService.getOAuth2Service().getAccessToken(code)`
6. 成功后把 `openId` 写入 Redis，TTL 300 秒
7. 如果微信抛出：
   - `40163`：说明这个 `code` 已被使用
   - `40029`：说明这个 `code` 已过期/非法
8. 这时服务统一抛出 `WxOAuthCodeException`
9. Controller 捕获后返回统一的 `ApiResponse.error(...)`
10. 前端根据 `message` 给出更友好的弹窗提示

这个实现避免了把微信 SDK 的异常栈直接泄露给前端，同时也避免了同一页面短时间多次提交时的重复兑换问题。

#### 大白话回答

第一次提交时，我让系统先去微信拿一次“用户身份证号”，然后偷偷记到 Redis 里。用户马上又点第二次时，就不用再跑去找微信问了，直接拿刚才记住的结果就行。如果这张“票”真的废了，我也不会再让页面弹“未知错误”，而是直接告诉用户：请从公众号菜单重新进来。

### 8. 面试官如果问：你是如何测试这个模块的？

#### 技术版回答

我没有只做人工联调，而是把可重复的风险点固化成单元测试：

- `WxOAuthCodeServiceTest`
  - 测缓存命中
  - 测首次成功缓存
  - 测微信返回 `40163` 后从缓存恢复
  - 测微信返回 `40163` 且无缓存时抛业务异常

- `EccControllerTest`
  - 测 Controller 能否把 `WxOAuthCodeException` 转成前端可消费的错误响应
  - 验证遇到授权重复问题时，不会继续进入作文批改服务，避免做无效调用

最后再跑整仓 `mvn test`，保证这次修复没有破坏原有模块。

#### 大白话回答

我不只是“自己拿手机点了一下试试”，而是专门写了自动化测试去模拟：

- code 正常用一次
- code 被重复用
- code 已经过期

这样以后就算我再改授权逻辑，也能第一时间知道有没有把这个坑重新引回来。

### 9. 这个点相关的八股文

#### 9.1 为什么微信 OAuth `code` 不能重复使用？

这是 OAuth 授权码模式本身的安全设计。授权码是一次性票据，防止中间人截获后无限次复用。

**大白话：**
它就像电影院一次性的入场二维码，扫一次就废，不能拿着同一张票反复进场。

#### 9.2 为什么这里适合用 Redis 做短期缓存？

- `code -> openId` 的映射生命周期短
- 访问频率高但数据量小
- Redis 读写快，天然适合做几分钟级别的临时授权缓存

**大白话：**
这种数据就像“刚刚查过一次、马上还会再用”的便签纸，放 Redis 最合适，不需要次次都再问微信。

#### 9.3 为什么不能把第三方异常原样抛给前端？

因为第三方异常：

- 不稳定
- 不友好
- 可能暴露技术细节

而前端需要的是可理解、可操作的提示语。

所以最佳实践是：

- 后端把外部错误码翻译成内部可控的业务语义
- 前端只展示用户能理解的操作提示

**大白话：**
用户不需要看“40163 code been used”这种行话，他需要知道的是：
“你这个页面授权已经失效了，请重新从公众号菜单进去。”

### 10. 这次修复后的最终价值

这次修复的价值，不只是“把异常压下去”，而是把这个模块从“碰运气能跑”提升到了“有清晰授权边界和用户反馈”的状态：

- 同一页面短时间二次提交不再直接炸掉
- 真的过期时也有明确的用户提示
- Controller 不再直接耦合微信 OAuth 细节
- 授权解析逻辑被统一复用
- 自动化测试把这个坑固定住了

一句话总结：

> 我这次修的不是一个提示框，而是把“微信授权票据只能用一次”这个协议级坑，收敛成了一个可复用、可测试、可解释的授权模块。

大白话版本：

> 以前这个问题像“用户点第二次就随机炸一下”，现在变成了“系统知道为什么炸，也知道怎么优雅地告诉用户下一步怎么做”。
